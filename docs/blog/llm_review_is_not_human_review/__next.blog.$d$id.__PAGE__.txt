3:T93b,<p>LLM Review is, simply, having an LLM review and critique a piece of written work. LLM Review is a critical and necessary phase of agentic workflows, but it is not a substitute for human review (in programming, code review), because these achieve different aims.</p>
<p>LLM Review against an LLM output helps refine that output towards a set of examples. Without providing examples, those will usually be some  net &quot;average&quot; of the type of thing it&#39;s reviewing. With examples, including examples from the current code base or similar documents from the organization, the review will move more towards that average. This is good in that it keeps bad things from getting worse, but left alone it blocks good things from getting better.</p>
<p>LLM Review applied to human written content is a bit different- while it has the same underlying effects, the benefit from LLM review suggestions is finding patterns a human may have missed. This &quot;blind spot&quot; search can be a welcome backstop, but must be considered in balance with an LLM&#39;s push towards the average.</p>
<p>Human review is entirely different. The purpose of human review is to build a shared understanding. Human to human review is always a two-way street, with the reviewer learning from the author&#39;s writing, and the author learning what is and is not convincing to the reviewer.</p>
<p>LLMs don&#39;t learn from conducting a review. They only &quot;learn&quot; (be it through training or RAG) after the work product has been accepted and committed in full. There&#39;s never a meeting of minds, or shared understanding, when an LLM is involved in creating a document.</p>
<p>LLM review is certainly a valuable, albeit at times sharp and harmful, tool in the modern digital world. It does not, cannot, and should not be used as a replacement for humans reviewing each other&#39;s work.</p>
