You are building an enrichment dataset that conforms to the Enrichment Schema,
to accompany <itinerary-file>.yaml.

Rules:
- This is a PURE display layer. Never restate booking facts (times,
  confirmation numbers, prices). Anything here is optional and may fail to
  load, so put nothing load-bearing in it.
- Join keys must match the itinerary EXACTLY:
    page_cards[].city   = a city in the trip
    flights[].flight    = "<airline_code><flight_number>" (e.g. "DL1")
    hotels[].hotel      = the hotel's name in the itinerary
    activities[].event  = the event's title in the itinerary
- Images: provide image_query SEARCH STRINGS, not hardcoded image URLs. For
  places, also provide wikipedia { title, url, summary_api } so the build-time
  wiki cache (see "Wikipedia cache" below) can pull a license-clean lead image
  and first paragraph.
- Flights: airline_status_url (airline's own page) + tracker_url on FlightAware
  (https://flightaware.com/live/flight/<ICAO><number>; DL->DAL, OS->AUA,
  BA->BAW, AA->AAL, UA->UAL, AF->AFR, LH->DLH).
- Hotels: for confirmed stays add website/address/phone/map_url; for to_book
  stays add a booking_search link to my preferred chain (IHG) for that city
  and dates.
- Activities: add a blurb, an official_url when one exists, and a
  destination_ref linking to a destinations[] entry with a Wikipedia card.
  (The renderer adds a Brave search booking link automatically.)
- Wikipedia cache: wikipedia.title feeds a BUILD-TIME fetch, not a runtime
  call. Use the EXACT article title from the page URL (underscores for spaces,
  URL-encoded: "Schönbrunn_Palace" -> "Sch%C3%B6nbrunn_Palace"); the cache keys
  on it verbatim. After writing the dataset, run
  `node scripts/wiki-cache.ts <id>` (the trip's directory name under trips/) to
  fetch each page_cards[]/destinations[] REST summary and write the committed
  trips/<id>/wiki-cache.json (lead image + extract + article URL). Re-run it
  whenever you add or change a wikipedia.title, then commit the cache. Render
  reads ONLY that cache, so the build stays offline and deterministic; a title
  with no entry simply renders no image.
- Trip prep (optional, advisory): meta.travelers records each traveler's
  passport_country and document STATUS only (type/status/valid_until). NEVER put
  raw passport, visa, or ETA reference numbers here - this file is deployed.
  trip_prep is a failure-silent "before you go" checklist: each item that states a
  rule must link (url) to the AUTHORITATIVE government source and must not restate
  the rule as settled fact. Stamp trip_prep.checked_on with the verification date.
  Do NOT hand-author entry/customs specifics in this prompt - run the `trip-prep`
  skill, which researches current requirements for the trip's countries and the
  travelers' nationalities, then writes meta.travelers + trip_prep.
